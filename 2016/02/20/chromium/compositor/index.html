Chromium Compositor(cc)

Layer Architech

Process Architecture

cc 的架构支持 single-process 的, multi-process，他建立在 LayerTreeHost 和 LayerTreeHostImpl 的基础之上，LayerTreeHost 表示逻辑上的 Tree, 而 LayerTreeHostImpl 完成实际的渲染操作。LayerTreeHost 和 LayerTreeHostImpl 的通信通过 Proxy 完成，如果 Proxy 的通信在线程内部完成 (通过 Proxy 的某个继承类 SingleThreadProxy), 他就是 single-process 的，如果是基于 IPC 的那么他就是多进程的。

class Proxy

类 Proxy 的功能是提供 LayerTreeHost 到 LayerTreeHostImpl 的通信功能，LayerTreeHost 通过调用 Proxy 的 API 向 LayerTreeHostImpl 发送指令。

Compositor 流程

compositor 的是“将所有 cc::Layers 组合起来完成 rasterizer 的过程，是 cc 最核心的流程，开启这个流程的接口是 LayerTreeHost::Composite 函数。文档将以此函数为例讲解 LayerTreeHost 和 LayerTreeHostImpl 的工作流程

Layer

cc::Layer 的构建

从使用者就读看 Chromium Compositor

//ui

ui/compositor 和 ui/aura 都是 Chromium Compositor 的重要用户

ui/compositor 的用户是 aura::WindowTreeHost (aura 是一个新的无窗口 (no HWND) 的窗口实现)。 aura::WindowTreeHost 代表着这个窗口系统的唯一窗口，它是所有虚拟窗口 aura::Window 的载体，aura::Window 可能是 ui::Layer (class Layer : public cc::LayerClient {..}) 的 owner，如果 aura::Window 需要独立渲染，那么他就会拥有一个独立的 ui::Layer。aura::WindowTreeHost 拥有一个 ui::Compositor 的对象，渲染时 aura::WindowTreeHost 自身的 LayerTree 交给 ui::Compositor 渲染即可。LayerTree 是根据 aura::Window Tree 的继承关系及是否拥有独立的 ui::Layer 构建的。这个的层次非常清楚

  • aura::WindowTreeHost, aura::Window： 负责管理窗口的层次关系及消息传递
  • ui::Compositor 和 ui::Layer: 负责将 ui::Layer 创建 cc::Layer 的映射关系并提供接口
  • cc::LayerTreeHost 和 cc::Layer: 完成具体渲染
