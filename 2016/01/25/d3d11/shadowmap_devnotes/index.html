Shadowmap Devnotes

Jan 25, 2016 | DirectX | Hits

Tricks

Light Project Matrix

SV_Position 的定点会在 Rasterizer 阶段进行一些特殊操作 ( W-Devide )，对于普通的定点(World Position), 这些本来依赖于 Render Pipeline 的操作需要我们自己来完成，具体的操作可以参考 Interactive Computer Graphics 和 Practical Rendering and Computation with D3D11 这两本书的内容。

Double Speed Model

在渲染模型的深度时，使用深度缓冲区并且将 ps shader 设置为 NULL， DirectX 能够更高效的完成任务 (Double Speed Model)

Depth Buffer 格式

将 Depth Buffer 作为 Shader Resource 使用时，需要使用特殊格式

Troubles

使用 Light Camera 渲染时，没有 Pixel Stage

在使用 Light Camera 渲染时，从 Visual Studio 的图形诊断看，Vertex Stage 是正常的，但是 PixelStage 就没有内容了。
原因是：对于 Light Camera 来说, 这些对象都是不可见的。

判断投影坐标是否正确

可以直接将深度绘制出来，不进行其他任何计算，看是否符合预期

对象投影错误

原因是传递的 Light Projection View Matrix 没有乘上物体的 World matrix, 这个错误在当前框架下比较容易犯。

深度计算错误

计算深度有两个地方，一个地方是在生成 shadowmap 时， 另外是在计算阴影时(需要 light_project_view * worldpos)。由于对 Render Pipeline，在计算shadowmap时，输出了 o.position.z / o.position.w 作为深度（SV_Position 变量），实际上 o.position.z 已经进行在 Rasterizer State 时完成过这个运算了。在计算阴影的时候总会觉的 light_pvpos.z / light_pvpos.w / light_pvpos.w 才是正确的值，而实际只要除一次就好了。另外，对于 shadowmap 的运算最好关闭 pixel shader 只输出 depth buffer, 这样的效率最高，使用 depth buffer 作为 shadowmap

HLSL 函数 SampleCmpLevelZero

SampleCmpLevelZero 函数会做 PCF Sampler 并于给定值比较，不过他依赖于 SamplerState（需要指定比较函数，默认是 Never, 此外 Filter 要使用 COMPARISION_FILTER，否则创建出来的 SamplerState 会不符合预期）。

在使用这个函数的过程中还遇到另外一个问题是 SamplerState 无法在 HLSL 程序中初始化，这其实与 hlsl 的机制有关，如果使用 const 的定义，应该问题就解决了。

DepthMap 不过 RenderTarget 的效果好

目前使用 R32 的 DepthBuffer 作为输入，不如 kRGBAn32 格式的 RenderTarget 的效果好

不要在 Vertex Stage 阶段计算深度

如果在 Vertex Stage 阶段计算深度，它的值不会进行插值计算，在 PixelStage 阶段会有较大的偏差，效果很差。
