Build Chromium with GN

Build Chromium with GN

Trouble Shooting

msvcr120d.dll 无法找到

build/gyp_chromium 会执行 build/vs_toolchain.py 的 copy_runtime 方法，将运行库拷贝过去，但是我们使用的 gn， 似乎目前还没有这个功能。
有一点需要注意，自己手动copy的话容易犯错，最好使用 gyp_chromium生成好目录后 copy 过去比较好，注意到一点是如果你的机器是 64 为机器那么应该 copy Debug_x64 的.

azer::InitApp() 无法正常工作

azer::InitApp 完成 CommandLine 和 AtExitManager 的初始化工作，在main()当中调用了 appinit 之后发现 render.dll 当中依赖于 AtExitManager 的 MessageLoop 仍然无法工作。当如果将 appinit 由 source_set 更改为 AZER_EXPORT 则没有问题。通过调试器发现自始至终程序都没有加载base.dll, 因此判断 is_component_build=false 导致生成的 base 是 static_libarary 进一步导致问题 AtExitManager 在 render.dll 和 可执行程序当中各有一个实例，在一个地方初始化无法完成。

yasm CreateProcess 失败

这是由于 environment.x64当中的环境变量设置错误，导致无法正确找到cl.exe，将x86_amd64 拷贝到 amd64就可以了
此外，即使找到了由于找不到对应的 mspdb120.dll 还是会出现错误，直接将depot_tools\win_toolchain\vs2013_files\VC\bin下的mspdb拷贝过来就好了，此dll是x86和x64共享的。
environment.x64 和 environment.x32 是如何获得这些路径的呢？它实际上是 gn 的一个目标，定义在build/config/win/visual_studio_version.gni文件当中，它通过一个脚本build/vs_toolchain.py来获得 visual studio 的相关信息。
通过运行： python build\vs_toolchain.py get_toolchain_dir scope 可以看到当前使用的 visual studio 的变量设置，从 build\vs_toolchain.py的代码可以看到其中细节，这些变量全部保存在了 //build/win_toolchain.json 当中，更改他们就可以了，也可以通过环境变量更改；

此外 depot_tools/win_tools/package_from_installed.py 提供了从已经安装的部分 copy 文件的方式，需要注意的是脚本内部制定了 Visual studio 2013 的安装目录（C:\Program Files (x86)\Microsoft Visual Studio 12.0），如果与预期不一致，可以修改再运行。
使用 gclient runhooks 更新 win_toolchain 时，目标目录在 depot_tools/win_tools/data.json 下定义

third_party nss 编译失败

发现问题出现在 .asm 文件不会生成编译代码上，原因是 .asm 的文件并没有识别成 asm 代码，将文件copy一份并命名为 .S，同时在 BUILD.gn 当中修改即可。

libcmt

连接过程中，可能回报* already defined in MSVCRd.lib 等，base/allocator 封装了 tcmalloc 并将它伪装成了 libcmt.lib，所有的executable都依赖于这个库，将这个库的依赖移除即可
此外：如果发现项目目录下没有 BUILD.gn, 那么几有可能是由于 gn 才 .gn定义的 secondary 目录下，目前是 //build/secondary

gin/function_template.h 编译错误

visual studio 2013需要升级到update 3 以上

flex & bison internal error

运行 third_party/cygwin/bin/flex 或者 third_party/cygwin/bin/bison 时，报错 internal error, 通常是由于他们依赖的m4.exe无法找到/usr/shared../m4sugar导致的，third_party/cygwin/setup_mount.bat 提供了解决方法，直接运行它就可以了

编译时打印 includes 文件的路径

修改 //build/toolchain/win/BUILD.gn 去掉 /showIncludes

连接时报错 atlthunk.lib 无法找到

只有当 is_visual_studio_express 为 ture 时，需要此文件（//build/config/win/BUILD.gn）, is_visual_studio_express 通过变量 visual_studio_version 赋值(build/config/win/visual_studio_version.gni:
is_visual_studio_express = (visual_studio_version == “2013e”)),
而变量 visual_studio_version 通过 //build/win_toolchain.json 定义，修改它就好了

很多 undfined symbols

遇到这种问题，可以简单的对照 gyp_chromium 和 gn 生成的link命令就能找到答案了

在gn gen out/dbg下出现了64

这是正常的，由于跟多 BUILD.gn 生成的是工具(例如 yasm), 高效的方式是生成的binary与build的机器的CPU arch一致，因此会在root_build_dir下包含一个64的目录。
如何做到呢， 可以参考 //BUILD.gn

Docker 快速介绍

本文档主要目的是说明 docker 的最简单也是最核心的功能。

Docker 的最重要功能是系统镜像(包括系统内核、系统库、系统程序及数据)，例如服务端程序依赖于 NLP 模块、NLP 词典以及对应.so文件。 不过的服务程序可能不便，但这几个基本模块都需要。在这种情形下，我们可以按照当前内核基础上创建一个镜像，并在此镜像内部加入 NLP 的 so 及字典文件。

接下来我们将使用最简单的代码及依赖关系说明如何 Docker 进行快速部署。

启动

Container 基本是以程序的生命周期来进行的，如 docker -r -i … /bin/bash， 在 /bin/bash 退出时(如运行 exit 命令), 则整个 docker 都退出了。 Container 也可以以类似 (nohup ./run.sh &) 的形式运行，及 daemon 形态。
