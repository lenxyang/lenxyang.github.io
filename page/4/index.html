Tessellation Devnotes

Tessellation (DevNotes)

Troubles Shooting

渲染 quad 时只能出现两个对角三角形，而非4个三角形

问题发生在插值函数上

开始是将 v2 的插值顺序写反了 (quad[3], quad[2])
注意：azer3d 采用的是右手坐标系

Terrain Hull Constants Shader 计算

Terrain 的 Constants Shader 计算是基于距离的计算（深度或者到摄像头的具体），其基本思想是距离越大分割越少，不过有一点需要注意：并非距离越远分割越少，而是超过一定距离之后就不在分割（或者仅仅分割一次）。

Terrain Quad Tessellation 分割不符合预期

观察到的现象：分割的数量没有规律，似乎在近处少，处在某个地方是分割的最多。为了修复这个问题尝试使用 “固定点” 代替 eyepos 作为距离计算的基点，发现分割似乎仍然与 eyepos 有关。后来通过 Visual studio 的调试工具观察，发现某些常量的值异常。

最终确定是因为 HLSL 对全部变量不支持操作的， 将四个全局变量移动到函数内部，结果符合预期.

避免龟裂

从直观上将只要保证任意一条边的分割是一致的就可以避免龟裂(边可能被多个三角形共享)，做到这点并不困难只要根据这个边的两个顶点计算 Hull Constants Shader 就可以保证了。

不过除此之外，还有一些其他隐含约束需要考虑，以 Quad 为例，他的顶点的顺序是在 IndexBuffer 当中指定的，而 Hull Constants Shader 的边的顺序是隐含约定的

如果计算边的分割使用的错误的顶点的话也会出问题

  • Hardware Accelerated Displacement Mapping for Image Based Rendering

GPU Server-Client Mode

GPU Server-Client

单进程初始化gpu::GLInProcessContext

要理解Server-Client结构，最好的方式能够找到Server及Client创建并建立联系的代码，在//gpu当中类GLInProcessContext是单进程渲染的gpu初始化类，当//gpu的不进行快进程调用时，此类将负责初始化，理解他是理解//gpu的一个很好的开始

GLInProcessContext类本身只定义了接口已经一个静态函数，这是为了尽量避免外部API的更改。GLInProcessContext的所有功能都有在.cc文件当中的Impl类来完成。从类的成员能够看出这个类完成的功能。

  • context_lostcallback 这是为了应对 ContextLost 而设立的响应机制
  • GLES2CmdHelper指针，
  • GLES2Implementationd的指针，定义了接口，所有的GL的调用都将通过它完成。
  • commandbuffer, 这是接受gpu请求的服务，是gpu c-s结构当中的s

GLInProcessContextImpl的接口一望而知，我们主要关注他的初始化过程。

建立连接

对象commandbuffer(类型InProcessCommandBuffer)是gpu的Service-Side, 而CommandBufferHelper是Client-Side，但然GLES2Implementation自然属于Client-Side，不过他们发起的通信都是通过CommandBufferHelper来完成的。 双发的通信是通过SharedMemory实现的RingBuffer做的，Service-Side会创建一个名为TransferBufferManager的对象，gleshelper在创建过程中能够通过commandbuffer获得该对象的指针并用它来创建TransferBuffer。

GPU Service-Side

我们仍然以GLInProcessContext为入手点，它创建了一个InProcessCommandBuffer对象并初始化它。

InProcessCommandBuffer

InProcessCommandBuffer的运行线程是单独创建，即使是初始化过程也在gpu进行中完成（虽然是同步方式调用）。这可能是因为GL的调用不一定是线程安全的，以Direct3D来说，D3D11DeviceContext就不能跨线程。

QueueTask将函数InProcessCommandBuffer::InitializeOnGpuThread运行，它完成CommandBuffer的真正的初始化工作，包括

  • 创建TransferBufferManager
  • 创建CommandBufferService并将设置接受消息的函数PumpCommands和ContextLoadCallback
  • 创建GLES2Decoder和GpuScheduler
  • 创建gfx::Surface和gfx::Context

TransferBufferManager

TransferBuffer(//gpu/command_buffer/client/transfer_buffer.h)和TransferBufferManager(//gpu/command_buffer/server/transfer_buffer_manager.h)封装了Client-Server的通信细节，实际是通过shared memory形式的ring buffer进行通信的。

gfx::Surface and gfx::GLContext

这两个类是真正的GL绘制命令，但从代码上也能够看出，一个进程当中可能会创建多个gfx::GLContext和gfx::Surface，这在一般的GL程序中并不常见，我们需要分析一下这段代码的逻辑及用途。
gfx::GLContext的创建过程比较直接，按照指定的实现创建对应类型的GLContext，类型有可能是EGL， WGL, MESA多种类型，Surface与之详细也有多种类型。
这两个对象创建过程中用到了了ShareGroup, 具体可以参考文档[ui/gl]。

GLES2Decoder and GpuSchedule

GpuSchedule是CommandBufferEngine的继承类，CommandBufferEngine的作用是操作CommandBuffer，在接受到请求以后由CommandBufferEngine进行操作；接下来是GLES2Decoder，他是CommonDecoder的子类，他完成的功能是将从CommandBuffer当中读出（读出的操作通过CommandBufferEngine完成）的命令解码并执行。GLES2Decoder的解码部分代码应该是简单逻辑，执行部分这是直接转发调用到传递给他的gfx::GLContext上。

GPU C-S Communication

  • ShareMemory的id如何交换
  • CommandBuffer的读取和写入

Build Chromium with GN

Build Chromium with GN

Trouble Shooting

msvcr120d.dll 无法找到

build/gyp_chromium 会执行 build/vs_toolchain.py 的 copy_runtime 方法，将运行库拷贝过去，但是我们使用的 gn， 似乎目前还没有这个功能。
有一点需要注意，自己手动copy的话容易犯错，最好使用 gyp_chromium生成好目录后 copy 过去比较好，注意到一点是如果你的机器是 64 为机器那么应该 copy Debug_x64 的.

azer::InitApp() 无法正常工作

azer::InitApp 完成 CommandLine 和 AtExitManager 的初始化工作，在main()当中调用了 appinit 之后发现 render.dll 当中依赖于 AtExitManager 的 MessageLoop 仍然无法工作。当如果将 appinit 由 source_set 更改为 AZER_EXPORT 则没有问题。通过调试器发现自始至终程序都没有加载base.dll, 因此判断 is_component_build=false 导致生成的 base 是 static_libarary 进一步导致问题 AtExitManager 在 render.dll 和 可执行程序当中各有一个实例，在一个地方初始化无法完成。

yasm CreateProcess 失败

这是由于 environment.x64当中的环境变量设置错误，导致无法正确找到cl.exe，将x86_amd64 拷贝到 amd64就可以了
此外，即使找到了由于找不到对应的 mspdb120.dll 还是会出现错误，直接将depot_tools\win_toolchain\vs2013_files\VC\bin下的mspdb拷贝过来就好了，此dll是x86和x64共享的。
environment.x64 和 environment.x32 是如何获得这些路径的呢？它实际上是 gn 的一个目标，定义在build/config/win/visual_studio_version.gni文件当中，它通过一个脚本build/vs_toolchain.py来获得 visual studio 的相关信息。
通过运行： python build\vs_toolchain.py get_toolchain_dir scope 可以看到当前使用的 visual studio 的变量设置，从 build\vs_toolchain.py的代码可以看到其中细节，这些变量全部保存在了 //build/win_toolchain.json 当中，更改他们就可以了，也可以通过环境变量更改；

此外 depot_tools/win_tools/package_from_installed.py 提供了从已经安装的部分 copy 文件的方式，需要注意的是脚本内部制定了 Visual studio 2013 的安装目录（C:\Program Files (x86)\Microsoft Visual Studio 12.0），如果与预期不一致，可以修改再运行。
使用 gclient runhooks 更新 win_toolchain 时，目标目录在 depot_tools/win_tools/data.json 下定义

third_party nss 编译失败

发现问题出现在 .asm 文件不会生成编译代码上，原因是 .asm 的文件并没有识别成 asm 代码，将文件copy一份并命名为 .S，同时在 BUILD.gn 当中修改即可。

libcmt

gin/function_template.h 编译错误

flex & bison internal error

运行 third_party/cygwin/bin/flex 或者 third_party/cygwin/bin/bison 时，报错 internal error, 通常是由于他们依赖的m4.exe无法找到/usr/shared../m4sugar导致的，third_party/cygwin/setup_mount.bat 提供了解决方法，直接运行它就可以了

编译时打印 includes 文件的路径

修改 //build/toolchain/win/BUILD.gn 去掉 /showIncludes

连接时报错 atlthunk.lib 无法找到

只有当 is_visual_studio_express 为 ture 时，需要此文件（//build/config/win/BUILD.gn）, is_visual_studio_express 通过变量 visual_studio_version 赋值(build/config/win/visual_studio_version.gni:
is_visual_studio_express = (visual_studio_version == “2013e”)),
而变量 visual_studio_version 通过 //build/win_toolchain.json 定义，修改它就好了

很多 undfined symbols

遇到这种问题，可以简单的对照 gyp_chromium 和 gn 生成的link命令就能找到答案了

在gn gen out/dbg下出现了64

这是正常的，由于跟多 BUILD.gn 生成的是工具(例如 yasm), 高效的方式是生成的binary与build的机器的CPU arch一致，因此会在root_build_dir下包含一个64的目录。
如何做到呢， 可以参考 //BUILD.gn
